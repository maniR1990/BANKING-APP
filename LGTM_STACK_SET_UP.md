# LGTM Stack Setup & Implementation Guide

This document provides a detailed, step-by-step guide to setting up the LGTM Stack (**Loki, Grafana, Tempo, Mimir/Prometheus**) locally using Docker Compose for the current NestJS/Nx microservices architecture (Auth, Customer, Account, Gateway).

---

## 1. Architecture Overview & The "Shared Util" Question

### Is a Shared Util Good?
**Yes.** In an Nx monorepo containing multiple microservices, duplicating OpenTelemetry (OTel), Pino logging, and Prometheus configurations across `auth`, `customer`, and `account` applications is an anti-pattern. 

**Recommendation:** Create a shared Nx library (`libs/observability`). This library will export pre-configured NestJS modules for logging, tracing, and metrics. Each microservice will simply import this module in its root `app.module.ts`.

### Target Folder Structure
```text
.
├── apps/
│   ├── account/
│   ├── auth/
│   └── customer/
├── libs/
│   └── observability/               <-- NEW: Shared library
│       ├── src/
│       │   ├── observability.module.ts
│       │   ├── logger.config.ts     (Pino setup with OTel trace_id injection)
│       │   └── tracing.ts           (OTel SDK initialization)
├── monitoring/                      <-- NEW: Infrastructure config
│   ├── docker-compose.lgtm.yml      (Can be merged with root docker-compose.yml)
│   ├── grafana/
```

## 2. Infrastructure Setup (Docker Compose)

Add the following services to your `docker-compose.yml` (or run them via an included `docker-compose.lgtm.yml`). 

### Explicit Components You Must Not Miss
1. **OpenTelemetry Collector (OTel Collector):** Applications should not talk directly to Tempo or Prometheus. They talk to the OTel Collector, which acts as a router/buffer to ship data to the respective backends.
2. **Promtail (Log Shipper):** Instead of complex Docker daemon logging drivers, run Promtail as a container that mounts the Docker socket/volumes to scrape container logs and ship them to Loki.
3. **Grafana Provisioning:** Do not configure Data Sources manually in the UI every time Docker spins up. Mount provisioning YAML files so Grafana is ready immediately.

### Docker Commands
To start the stack alongside your databases and services:
```bash
docker-compose -f docker-compose.yml -f monitoring/docker-compose.lgtm.yml up -d
```

## 3. Configuration & Usage per Stack Component

### 1. Tempo (Distributed Tracing)
*   **Purpose:** Stores the distributed traces generated by your NestJS apps.
*   **Setup:** Mount `tempo-config.yaml` to configure receivers (OTLP gRPC/HTTP).
*   **Usage:** Traces are sent from the microservices -> OTel Collector -> Tempo.

### 2. Prometheus / Mimir (Metrics)
*   **Purpose:** Scrapes and stores time-series data (e.g., HTTP request duration, CPU usage).
*   **Setup:** Create `prometheus.yml` and define scrape jobs for each microservice (`banking_auth:3000/metrics`, etc.).
*   **Config Usage:** In NestJS, use `@willsoto/nestjs-prometheus` in the shared library to expose the `/metrics` endpoint.

### 3. Loki (Log Aggregation)
*   **Purpose:** Highly scalable log aggregation system.
*   **Setup:** Mount `loki-config.yaml`. Run a **Promtail** container alongside it to scrape Docker logs.
*   **Crucial Integration:** Your Pino logger in NestJS *must* log in JSON format and include the current OpenTelemetry `trace_id`. Promtail pushes this JSON to Loki.

### 4. OTel Collector
*   **Purpose:** Receives OTLP data from NestJS and exports it.
*   **Config Usage (`otel-collector-config.yaml`):**
    ```yaml
    receivers:
      otlp:
        protocols: { grpc: {}, http: {} }
    exporters:
      otlp/tempo:
        endpoint: tempo:4317 # Ships traces to tempo
      prometheus:
        endpoint: 0.0.0.0:8889 # Exposes metrics for prometheus to scrape
    service:
      pipelines:
        traces:
          receivers: [otlp]
          exporters: [otlp/tempo]
        metrics:
          receivers: [otlp]
          exporters: [prometheus]
    ```

### 5. Grafana (Visualization UI)
*   **Setup:** Mount `datasources.yml` in `/etc/grafana/provisioning/datasources`.
*   **Config Usage:** Define Loki, Tempo, and Prometheus as data sources.
*   **The "Magic" Link:** In the Loki data source configuration, you *must* add a `derivedFields` mapping. This tells Grafana: "When you see `trace_id` in a Loki JSON log, make it a clickable button that links directly to the Tempo trace."

## 4. Application Level Implementation (NestJS)

### Commands to scaffold the shared library
```bash
pnpm nx generate @nx/nest:library observability --directory=libs/observability
pnpm install @opentelemetry/sdk-node @opentelemetry/auto-instrumentations-node @opentelemetry/exporter-trace-otlp-grpc
pnpm install nestjs-pino pino-http
pnpm install @willsoto/nestjs-prometheus prom-client
```

### Shared `tracing.ts` Usage
Before NestJS bootstraps (`main.ts`), you must initialize the OTel Node SDK:
```typescript
// libs/observability/src/tracing.ts
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-grpc';

export const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter({
    url: 'http://otel-collector:4317', // Docker internal hostname
  }),
  instrumentations: [getNodeAutoInstrumentations()],
});

sdk.start();
```
*Note: Include this file at the very top of `apps/auth/src/main.ts`, etc.*

## 5. Developer Workflow in Grafana

Once fully implemented, your workflow for debugging an issue will be:
1.  Open Grafana (`http://localhost:3000`).
2.  Navigate to the **Explore** tab and select the **Loki** data source.
3.  Query logs for an error: `{container="banking_auth"} |= "error"`.
4.  Expand the JSON log line. You will see a `trace_id` field with a **Tempo** button next to it.
5.  Click the button. Grafana switches to the Tempo data source and displays a Gantt chart showing the request hitting NGINX Gateway -> Auth Service -> PostgresDB, highlighting exactly which span (function/request) took too long or failed.
